1. How to Fork the Repository

Forking creates your own copy of the Gerrit Monitor repository, allowing you to customize and run it independently.

Step-by-Step Instructions

1 Navigate to the Repository

Go to the original Gerrit Monitor repository on GitHub:

https://github.com/Nikhil-Ashoka/Gerrit-Monitor

2 Click the Fork Button

In the top-right corner of the repository page, click the Fork button.

3 Select Your Account

Choose where to fork the repository (your personal account or an organization you have access to).

4 Wait for Fork Completion

GitHub will create a copy of the repository under your account. This usually takes a few seconds.

5 Clone Your Fork Locally

Clone your forked repository to your local machine:

git clone https://github.com/YOUR-USERNAME/Gerrit-Monitor.git
cd Gerrit-Monitor
✅ Success! You now have your own copy of the Gerrit Monitor repository that you can customize and configure.

Enable GitHub Actions

1 Go to Actions Tab

In your forked repository, click on the Actions tab.

2 Enable Workflows

If prompted, click "I understand my workflows, go ahead and enable them" to activate the automation workflows.

📝 Note: GitHub Actions are free for public repositories and include 2,000 minutes per month for private repositories on free accounts.

2. Set Up Slack for Automation

Slack integration enables daily automated notifications of Gerrit activity directly to your team's Slack channel.

Create a Slack Incoming Webhook

1 Go to Slack API

Visit https://api.slack.com/apps

2 Create New App

Click "Create New App" and select "From scratch"

  • App Name: Gerrit Monitor (or your preferred name)
  • Workspace: Select your Slack workspace
3 Enable Incoming Webhooks

In your app settings:

  • Click on "Incoming Webhooks" in the left sidebar
  • Toggle "Activate Incoming Webhooks" to On
4 Add Webhook to Workspace

Scroll down and click "Add New Webhook to Workspace"

  • Select the channel where you want notifications to appear
  • Click "Allow"
5 Copy Webhook URL

Copy the generated webhook URL. It will look like:

https://hooks.slack.com/services/T00000000/B00000000/XXXXXXXXXXXXXXXXXXXX

Configure Slack in GitHub Actions

1 Go to Repository Settings

In your forked repository, navigate to Settings → Secrets and variables → Actions

2 Add Repository Secret

Click "New repository secret" and add:

  • Name: SLACK_WEBHOOK_URL
  • Value: Paste your webhook URL
3 Save the Secret

Click "Add secret" to save.

Configure Slack for Local Testing (Optional)

If you want to test Slack notifications locally:

1 Create .env File

Copy the example environment file:

cp .env.example .env
2 Add Webhook URL

Edit .env and uncomment the Slack webhook line:

SLACK_WEBHOOK_URL=https://hooks.slack.com/services/YOUR/WEBHOOK/URL
⚠️ Security Warning: Never commit your .env file to Git. It's already included in .gitignore to prevent accidental commits.

Test Slack Integration

1 Trigger Daily Workflow Manually

Go to Actions → Daily Gerrit Monitor → Run workflow

2 Check Your Slack Channel

Within a few minutes, you should see a formatted notification in your selected Slack channel.

✅ Slack Setup Complete! Daily reports will now be automatically posted to your Slack channel at 9:00 AM UTC (2:30 PM IST).

3. Set Up Email for Automation

Email integration enables weekly automated reports sent via email to your team members.

Gmail Setup (Recommended)

Generate Gmail App Password

1 Enable 2-Step Verification

Go to Google Account Security

  • Click on "2-Step Verification"
  • Follow the prompts to enable it if not already enabled
2 Create App Password

Visit https://myaccount.google.com/apppasswords

  • Select "Mail" as the app
  • Select "Other (Custom name)" as the device
  • Enter "Gerrit Monitor" as the name
  • Click "Generate"
3 Copy App Password

Google will display a 16-character password. Copy it (remove any spaces).

Example: abcdabcdabcdabcd
📝 Important: Use the app password, NOT your regular Gmail password. Regular passwords will not work with SMTP authentication.

Configure Email in GitHub Actions

Add the following secrets to your repository:

1 Navigate to Secrets

Go to Settings → Secrets and variables → Actions

2 Add Email Secrets

Click "New repository secret" for each of the following:

Secret Name Value Description
SMTP_HOST smtp.gmail.com Gmail SMTP server
SMTP_PORT 587 SMTP port for TLS
SMTP_USER your-email@gmail.com Your Gmail address
SMTP_PASSWORD your-app-password 16-character app password
FROM_EMAIL your-email@gmail.com Sender email address
TO_EMAILS team@example.com,user@example.com Comma-separated recipient emails

Configure Email for Local Testing (Optional)

1 Edit .env File

Open your .env file and uncomment the email section:

SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=your-email@gmail.com
SMTP_PASSWORD=your-app-password
FROM_EMAIL=your-email@gmail.com
TO_EMAILS=recipient1@example.com,recipient2@example.com

Alternative SMTP Providers

You can use other email providers besides Gmail:

Provider SMTP Host Port Notes
Gmail smtp.gmail.com 587 Requires app password
Outlook smtp-mail.outlook.com 587 Use regular password
Yahoo smtp.mail.yahoo.com 587 Requires app password
SendGrid smtp.sendgrid.net 587 Use API key as password

Test Email Integration

1 Trigger Weekly Workflow Manually

Go to Actions → Weekly Gerrit Monitor → Run workflow

2 Check Your Email

Within a few minutes, check the recipient email inbox(es) for the HTML report.

3 Check Spam Folder

If you don't see the email, check your spam/junk folder and mark it as "Not Spam".

✅ Email Setup Complete! Weekly reports will now be automatically sent every Monday at 9:00 AM UTC (2:30 PM IST).

4. Subscribe to Desired Gerrit Repositories

Configure which Gerrit repositories you want to monitor by editing the config.json file.

Understanding config.json

The configuration file supports monitoring multiple Gerrit projects simultaneously. Each project can have its own settings.

Configuration Structure

1 Open config.json

Edit the config.json file in your repository:

{
  "gerrit_url": "https://gerrit.openbmc.org",
  "projects": [
    {
      "name": "openbmc/webui-vue",
      "check_days": 1,
      "max_results": 100
    },
    {
      "name": "openbmc/bmcweb",
      "check_days": 1,
      "max_results": 100
    }
  ]
}

Configuration Parameters

Parameter Description Example
gerrit_url Base URL of your Gerrit instance https://gerrit.openbmc.org
name Full project path in Gerrit openbmc/webui-vue
check_days Number of days to look back for changes 1 (daily), 7 (weekly)
max_results Maximum number of changes to fetch 100

Add a New Repository

1 Find Your Gerrit Project

Navigate to your Gerrit instance and find the project you want to monitor.

Example: https://gerrit.openbmc.org/q/project:openbmc/pldm

2 Extract Project Name

The project name is the part after project: in the URL.

Example: openbmc/pldm

3 Add to config.json

Add a new project object to the projects array:

{
  "gerrit_url": "https://gerrit.openbmc.org",
  "projects": [
    {
      "name": "openbmc/webui-vue",
      "check_days": 1,
      "max_results": 100
    },
    {
      "name": "openbmc/bmcweb",
      "check_days": 1,
      "max_results": 100
    },
    {
      "name": "openbmc/pldm",
      "check_days": 1,
      "max_results": 100
    }
  ]
}

Monitor Different Gerrit Instances

To monitor projects from a different Gerrit instance, change the gerrit_url:

{
  "gerrit_url": "https://gerrit.example.com",
  "projects": [
    {
      "name": "your-org/your-project",
      "check_days": 7,
      "max_results": 50
    }
  ]
}

Recommended Settings

Use Case check_days max_results Reason
Daily monitoring 1-2 100 Captures recent activity
Weekly monitoring 7 200 Comprehensive weekly summary
High-activity projects 1 200-500 Ensures all changes are captured
Low-activity projects 7-14 50 Reduces noise, captures trends

Remove a Repository

Simply delete the corresponding project object from the projects array in config.json.

Commit and Push Changes

1 Save config.json

Save your changes to the configuration file.

2 Commit Changes 
git add config.json
git commit -m "Update monitored Gerrit repositories"
3 Push to GitHub 
git push origin main
✅ Configuration Complete! Your workflows will now monitor the updated list of repositories.

Test Your Configuration

1 Test Locally

Run the monitor script locally to verify your configuration:

python monitor.py
2 Check Output

Review the generated GERRIT_DAILY_REPORT.md file to ensure all projects are included.

3 Trigger Workflow Manually

Go to Actions and manually trigger the daily or weekly workflow to test the automation.

Troubleshooting

Workflow Not Running

  • Verify GitHub Actions are enabled in your repository
  • Check that all required secrets are configured
  • Review the Actions tab for error messages

Slack Notifications Not Appearing

  • Verify the webhook URL is correct
  • Check that the webhook hasn't been revoked
  • Ensure the daily workflow is running (not the weekly one)

Emails Not Received

  • Check spam/junk folders
  • Verify SMTP credentials are correct
  • Ensure you're using an app password for Gmail
  • Check the weekly workflow logs for errors

Empty Reports

  • Verify the Gerrit project names are correct
  • Increase check_days if there's low activity
  • Check that the Gerrit instance is accessible
